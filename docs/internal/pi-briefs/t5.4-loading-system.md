## Part 1 — Skeleton primitives

New file `src/components/ui/Skeleton.tsx` exporting four components:

- `SkeletonLine`: a single line, `h-3.5 rounded-control bg-surface-2`. Accepts a `width?:
  string` prop (CSS width value, e.g. `"60%"`, `"120px"`) applied via inline `style`, defaulting
  to `"100%"`.
- `SkeletonBlock`: a rectangular block. Accepts `width?: string` and `height?: string` (inline
  style, both default `"100%"`), same `rounded-control bg-surface-2` treatment.
- `SkeletonPill`: `h-5 w-16 rounded-pill bg-surface-2` (matches the shape of the ui-kit's
  `Pill` component, for skeleton-ing status pills specifically).
- `SkeletonRow`: a horizontal flex row combining a small `SkeletonBlock` (avatar/icon-shaped,
  e.g. `h-8 w-8 rounded-control`) and a stacked pair of `SkeletonLine`s of different widths
  (e.g. 70% then 40%) with a gap — this is the generic "list row" skeleton shape.

All four get the shimmer treatment: apply Tailwind's existing `animate-shimmer` class (already
defined in `tailwind.config.js` from T5.1 — a 1.6s opacity sweep, 0.03→0.07→0.03 — do not
redefine it, do not add a new keyframe, just apply the class). Every component should accept
and forward a `className?: string` for the caller to add margin/layout utilities.

Also export a `SkeletonGate` helper (not a visual primitive — a tiny wrapper component):
`<SkeletonGate loading={boolean} skeleton={ReactNode}>{children}</SkeletonGate>` that renders
`skeleton` while `loading` is true and `children` once it's false, with a 200ms opacity
crossfade between them (plain CSS `transition-opacity duration-reveal`, no framer-motion
needed here — see the `FieldError.tsx` pattern from T5.2 for the CSS-only fade-in technique).
The plan's rule is "skeletons appear only if the query exceeds 150ms, and fade out 200ms as
real content fades in" — implement the 150ms delay INSIDE `SkeletonGate` itself (a
`setTimeout`-driven internal `showSkeleton` state that only flips true after 150ms of
`loading === true`, and flips back to false immediately when `loading` becomes false) so every
caller gets the anti-flash behavior for free without repeating the timer logic.

### `PipelineTracker.tsx`

Props:
```ts
interface RunProgress {
  current_node: string | null;
  completed_nodes: string[];
  node_timestamps: Record<string, { started_at?: string; finished_at?: string }>;
  fan_out: Record<string, { done: number; total: number }>;
}
interface PipelineTrackerProps {
  graph: "petition" | "rfe";
  status: "running" | "waiting_review" | "completed" | "failed";
  progress: RunProgress;
}
```
(`RunProgress`'s shape comes from the backend's `agent_runs.progress` column, written by
`backend/app/agents/runner.py`'s `_stream_with_progress` — read that function for the
authoritative shape; it's already built, reviewed, and live-verified, you're only consuming it.)

Pick the topology array from `graph` (`PETITION_TOPOLOGY` or `RFE_TOPOLOGY`). Render it as a
horizontal stepper, one item per node, connected by a thin horizontal line (`border-t
border-border`, or a `bg-border h-px flex-1` divider between items — your call, keep it simple).

**Per-node state derivation** (compute this exactly, don't improvise different rules):
- `"failed"` — `status === "failed"` AND `node.key === progress.current_node`.
- `"done"` — `node.key` is in `progress.completed_nodes` (and not the failed case above).
- `"gate-waiting"` — `node.isGate` AND `node.key === progress.current_node` AND
  `status === "waiting_review"`.
- `"active"` — `node.key === progress.current_node` and none of the above matched (i.e. a
  non-gate node currently running, or a gate node reached but not yet in `waiting_review` —
  edge case, treat as active).
- `"pending"` — none of the above (not yet reached).

**Per-state rendering:**
- `done`: a small filled circle with a check mark (`lucide-react`'s `Check`, size ~12), `--met`
  colored (`text-met`, `bg-met/10` circle fill or similar), no animation beyond a one-time
  200ms pop-in (a simple CSS `scale` transition on mount is enough — framer-motion optional,
  not required).
- `active`: a `--run` colored ring. Use a pulsing ring: an outer `absolute` span using
  Tailwind's built-in `animate-ping` class PLUS an arbitrary-value duration override
  `[animation-duration:1.8s]` (Tailwind supports arbitrary non-color CSS values like this fine
  — do not attempt to define a new keyframe in `tailwind.config.js`, you don't have write
  access to it anyway) layered over a solid inner dot, same two-span technique as
  `src/components/shell/RunIndicator.tsx` (`animate-ping` outer + solid inner), just recolored
  to `run` and the duration overridden. Beneath the node, render a live elapsed-time mono
  label (`text-[10px] font-mono text-text-faint`) computed as `now - node_timestamps[node.key
  ].started_at`, formatted `"Ns"` or `"Nm Ss"`. To make it live-update, use a `useState` tick
  driven by a 1s `setInterval` inside the component (clear on unmount) — a single shared
  interval for the whole tracker is fine, you don't need one per node.
  If `node.isFanOut` and `progress.fan_out[node.key]` exists, render `"{done}/{total}"` next to
  the label instead of (or alongside) the elapsed timer — e.g. `"6/8"` — plus a small ring
  showing done/total as a fraction (an SVG circle with `stroke-dasharray` sized by the
  fraction is fine; keep it simple, a filled arc is enough, don't over-engineer this).
- `gate-waiting`: render the node as a diamond (a square `div` rotated 45deg via
  `rotate-45`, `--partial` colored border/fill — `border-partial`/`bg-partial/10`) instead of a
  circle, to visually read as "waiting on a human" per the plan. No pulse animation needed
  (a gate isn't "in progress," it's paused).
- `pending`: a hollow circle, `border border-text-faint`, no fill, `text-text-faint` label.
- `failed`: a filled `--gap` circle with an `X` (`lucide-react`, size ~12).

Label text (`node.label`) sits below/beside each marker, `text-[11px]` uppercase tracking-wide,
color matches the node's state color (dim/faint for pending, full tone otherwise).
